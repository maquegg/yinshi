## 2. 业务上下文

### 2.1 背景

临期商品需要实现"一品多位"销售，同一 EAN 商品同时存在：

- **正价商品**（非临期批次）
- **临期商品**（临期批次，特惠价）

两个商品的库存链路相互独立，销售系统需要分别管理，并通过不同路径与 WMS/库存中心交互。

### 2.2 目标

1. 明确销售系统库存的**增加**与**减少**机制
2. 梳理正价与临期商品的**库存链路差异**
3. 说明 C 端库存展示与**下单后数据传递 WMS** 的逻辑

### 2.3 涉及范围


| 涉及系统 | 职责              | 所属团队 |
| ---- | --------------- | ---- |
| 销售系统 | 库存管理、预占、扣减、C端展示 | 待定   |
| 库存中心 | 正价库存同步          | 待定   |
| WMS  | 临期标记、库存同步、履约执行  | 待定   |
| OMS  | 派单任务生成          | 待定   |
| 订单系统 | 订单创建、状态管理       | 待定   |


### 2.4 不做的事情（Out of Scope）

- 临期商品的促销定价策略（由促销中心负责）
- 正价商品的库存分配优先级逻辑
- 跨仓库调拨逻辑
- 库存对账与盘点
- 临期商品的退货处理（仅退款）
- 效期预警系统的实现（WMS侧）
- 人工手动上下架临期商品（系统自动处理）

## 5. 业务规则

### 5.1 状态机

> 库存状态转移（非用户操作状态机，为后台系统间交互逻辑）


| 编号  | 起始状态 | 触发动作    | 目标状态 | 执行角色 | 前置条件    | 副作用       |
| --- | ---- | ------- | ---- | ---- | ------- | --------- |
| S1  | 无    | 库存中心同步  | 可售   | 库存中心 | -       | 更新可售库存    |
| S2  | 可售   | 用户下单预占  | 已预占  | 销售系统 | 库存>0    | 锁定批次库存    |
| S3  | 已预占  | 支付成功    | 待履约  | 订单系统 | 预占成功    | 下发WMS     |
| S4  | 待履约  | WMS完成履约 | 已完成  | WMS  | 拣货校验通过  | 扣减库存      |
| S5  | 已预占  | 订单取消    | 已释放  | 销售系统 | 用户取消/超时 | 释放库存回写WMS |
| S6  | 已预占  | 履约失败    | 已释放  | WMS  | 缺货等     | 释放库存      |


> **终态**：已完成、已释放

### 5.2 计算规则


| 规则名      | 公式/逻辑          | 适用场景         | 示例                                               |
| -------- | -------------- | ------------ | ------------------------------------------------ |
| FIFO批次分配 | 按效期从早到晚排序，依次预占 | 用户下单多件多批次    | 购买3件，批次效期[6-1, 6-15, 6-20] → 预占[6-1×2件, 6-15×1件] |
| 多件多批次    | 订单数量分散到多个最早期批次 | 订单数量 > 单批次数量 | 购买5件，单批次最大100件 → 分配至多个批次                         |
| 部分预占失败   | 库存不足时，整个订单预占失败 | 预占失败         | 预占3件但只有2件 → 订单取消                                 |


### 5.3 通用约束

- **库存隔离**：正价与临期商品库存完全隔离，独立计数、独立预占，互不影响
- **库存扣减时机**：必须收到 WMS 履约完成通知后才能扣减
- **批次锁定**：预占时必须锁定具体批次，不能只锁定 SKU 总数
- **自动上下架**：临期库存=0自动下架；临期库存>0自动上架
- **效期校验**：履约时 WMS 校验批次效期，确保发出的商品效期符合要求

## 6. 功能需求明细

### 6.1 库存增加

#### US-1-1: 正价商品库存同步

**角色**: 作为 销售系统
**目标**: 我想要 接收库存中心的正价库存同步
**价值**: 以便于 在 C 端展示可售库存

**所属端**: 销售系统
**前置条件**: 商品已在 WMS 入库，库存中心有库存数据

**业务规则**:

- [新增] 正价库存同步来源为库存中心
- [新增] 正价商品不带批次信息
- [新增] 同步触发：商品入库后，库存中心主动推送至销售系统
- [Q1] 同步频率：**实时推送**

**页面/交互描述**: 无需页面，纯后台数据同步

**接口**: 库存中心 → 销售系统（库存同步接口） → DEP-1

**同步数据字段**:


| 字段              | 类型       | 说明     |
| --------------- | -------- | ------ |
| sku_id          | String   | SKU ID |
| ean             | String   | EAN    |
| available_stock | Integer  | 可售库存数量 |
| update_time     | DateTime | 更新时间   |


**边界/异常**:

- 库存中心推送失败 → 销售系统记录异常日志，等待重试
- SKU ID 不存在 → 销售系统忽略该条同步

#### US-1-2: 临期商品库存同步

**角色**: 作为 销售系统
**目标**: 我想要 接收 WMS 直连的临期库存同步
**价值**: 以便于 管理临期商品库存

**所属端**: 销售系统
**前置条件**: WMS 已标记临期批次

**业务规则**:

- [新增] 临期库存同步来源为 WMS 直连
- [新增] 临期商品带批次信息（批次号、效期）
- [新增] 临期库存增加时，同步通知库存中心扣减正价库存（临期↑ = 正价↓）
- [Q2] 同步方式：**实时推送 + 按需拉取**（实时推，按需拉取）

**页面/交互描述**: 无需页面，纯后台数据同步

**接口**: WMS → 销售系统（临期库存同步接口） → DEP-2

**同步数据字段**:


| 字段             | 类型      | 说明     |
| -------------- | ------- | ------ |
| sku_id         | String  | SKU ID |
| ean            | String  | EAN    |
| batch_no       | String  | 批次号    |
| expiry_date    | Date    | 批次效期   |
| stock_quantity | Integer | 临期库存数量 |
| location_code  | String  | 储位信息   |


**边界/异常**:

- 临期库存同步失败 → 记录异常日志，等待重试
- 批次效期已过期 → WMS 不应同步至销售系统（已过期商品不能销售）

#### US-1-3: 临期商品自动创建与SKU映射

**角色**: 作为 销售系统
**目标**: 我想要 临期商品首次同步时自动创建SKU映射
**价值**: 以便于 减少人工运营成本

**所属端**: 销售系统
**前置条件**: 临期库存首次同步（US-1-2）

**业务规则**:

- [新增] 自动创建条件：临期库存首次同步时，临期商品不存在
- [新增] 继承规则：继承正价商品的商品信息（名称、图片、描述等）
- [新增] 价格初始化：初始价格 = 正价商品的当前售价
- [新增] SKU映射：临期商品SKU与正价商品SKU建立映射关系（同一EAN）

**页面/交互描述**: 无需页面，系统自动处理

**交互链**:

1. WMS 同步临期库存至销售系统
2. 销售系统判断：临期商品是否存在
3. 不存在 → 自动创建临期商品（含SKU映射），价格继承正价售价
4. 已存在 → 更新临期库存

**边界/异常**:

- 正价商品不存在 → 记录异常日志，人工介入处理
- 价格继承时正价商品无售价 → 默认价格为空，需运营手动设置

#### US-1-4: 临期商品自动上下架

**角色**: 作为 销售系统
**目标**: 我想要 根据临期库存状态自动上下架
**价值**: 以便于 用户只能购买有库存的临期商品

**所属端**: 销售系统
**前置条件**: 临期库存变化（US-1-2）

**业务规则**:

- [新增] 下架条件：临期库存 = 0 → 自动下架临期商品
- [新增] 上架条件：临期库存 > 0 → 自动上架临期商品
- [新增] 上下架为系统自动行为，无需人工操作

**页面/交互描述**: 无需页面，系统自动处理

**交互链**:

1. 销售系统收到临期库存同步
2. 判断临期库存数量
3. 库存=0 → 标记临期商品为"已下架"
4. 库存>0 → 标记临期商品为"上架"

**边界/异常**:

- 库存从0变为1 → 自动上架
- 库存从1变为0 → 自动下架
- 商品被人工手动下架后，库存从0变为>0时，**允许系统自动上架覆盖人工设置**[Q5]

### 6.2 库存减少

#### US-2-1: 用户下单预占库存

**角色**: 作为 销售系统
**目标**: 我想要 用户下单支付成功后执行库存预占
**价值**: 以便于 确保履约时有货可取

**所属端**: 订单系统触发，销售系统执行
**前置条件**: 临期商品已上架（US-1-4），用户下单且支付成功

**业务规则**:

- [新增] 预占触发：用户下单并支付成功
- [新增] FIFO原则：按效期从早到晚排序，优先消耗最早效期的批次
- [新增] 多件多批次：如购买3件，库存有3个批次，按批次顺序依次预占
- [新增] 预占锁定：锁定对应批次的库存数量

**页面/交互描述**: 无需页面，订单系统调用

**交互链**:

1. 触发：用户选择商品（临期）、数量，点击下单
2. 订单系统创建订单，状态为"待履约"
3. 销售系统执行库存预占：按 FIFO 原则确定批次分配方案，锁定对应批次的库存数量
4. 生成预占指令（含批次分配），下发至 WMS
5. WMS 确认预占成功，返回预占结果
6. 订单状态更新为"已预占"

**接口**: 销售系统 → WMS（预占指令） → DEP-3

**预占指令数据字段**:


| 字段                              | 类型      | 说明     |
| ------------------------------- | ------- | ------ |
| order_id                        | String  | 订单号    |
| sku_id                          | String  | SKU ID |
| quantity                        | Integer | 数量     |
| batch_allocations               | Array   | 批次分配方案 |
| batch_allocations[].batch_no    | String  | 批次号    |
| batch_allocations[].quantity    | Integer | 预占数量   |
| batch_allocations[].expiry_date | Date    | 效期     |
| type                            | String  | 临期/正价  |


**边界/异常**:

- 预占失败（库存不足）→ 订单自动取消，释放预占
- 预占超时未确认 → 订单自动取消，**超时时间依赖订单实际取消时间**[Q3]
- 多件多批次时部分批次库存不足 → **整单预占失败**[Q6]

#### US-2-2: 履约完成扣减库存

**角色**: 作为 销售系统
**目标**: 我想要 收到 WMS 履约完成通知后扣减库存
**价值**: 以便于 完成库存闭环

**所属端**: 销售系统
**前置条件**: 订单已预占（US-2-1），WMS 完成拣货、出库

**业务规则**:

- [新增] 扣减触发：WMS 完成拣货、出库
- [新增] 扣减依据：履约完成通知中的批次扣减明细
- [新增] 扣减数量：按实际拣货的批次和数量扣减

**页面/交互描述**: 无需页面，纯后台处理

**交互链**:

1. WMS 收到 OMS 派单（含储位+批次信息）
2. 分拣员按指令到指定储位，拣取指定批次商品
3. PDA 扫码校验（储位码 + 批次标签）
4. 校验通过，包裹出库
5. WMS 推送履约完成通知至销售系统
6. 销售系统扣减对应批次库存

**接口**: WMS → 销售系统（履约完成通知） → DEP-4

**履约完成通知数据字段**:


| 字段                        | 类型       | 说明          |
| ------------------------- | -------- | ----------- |
| order_id                  | String   | 订单号         |
| sku_id                    | String   | SKU ID      |
| picked_batches            | Array    | 批次扣减明细      |
| picked_batches[].batch_no | String   | 批次号         |
| picked_batches[].quantity | Integer  | 扣减数量        |
| status                    | String   | 已完成/部分完成/失败 |
| completed_at              | DateTime | 履约完成时间      |


**边界/异常**:

- 拣货时发现批次库存不足 → **WMS回传实际拣货数量**[Q7]
- 履约完成通知丢失 → 销售系统未扣减库存，**需设计补偿机制（待确认具体方案）**[Q8]

#### US-2-3: 订单取消/退款释放库存

**角色**: 作为 销售系统
**目标**: 我想要 订单取消或退款时释放预占库存
**价值**: 以便于 库存回归可售状态

**所属端**: 销售系统
**前置条件**: 订单已预占但未履约（US-2-1）

**业务规则**:

- [新增] 释放触发：订单取消、退款、履约失败
- [新增] 释放内容：释放预占的批次库存
- [新增] 回写WMS：销售系统将释放的库存回写至 WMS
- [新增] 释放后库存变为0 → **触发自动下架**[Q9]

**页面/交互描述**: 无需页面，系统自动处理

**交互链**:

1. 订单取消/退款 → 释放预占库存
2. 履约失败（如缺货）→ 释放预占库存 → 触发异常处理
3. 销售系统将释放的库存回写至 WMS

**接口**: 销售系统 → WMS（库存释放） → DEP-5

**边界/异常**:

- 释放时 WMS 不可用 → 记录异常，等待重试

### 6.3 C端展示

#### US-3-1: 库存展示与隔离

**角色**: 作为 C端用户
**目标**: 我想要 看到正价和临期商品各自独立的库存
**价值**: 以便于 我能选择购买正价或临期，了解可售情况

**所属端**: C端
**前置条件**: 销售系统有库存数据

**业务规则**:

- [新增] 库存隔离：正价与临期商品库存完全隔离
- [新增] 独立计数：各自独立库存计数
- [新增] 独立预占：各自独立预占锁定
- [新增] 互不影响：临期售罄不影响正价，反之亦然

**页面/交互描述**:


| 场景   | 正价商品 | 临期商品  | 说明           |
| ---- | ---- | ----- | ------------ |
| 都有库存 | ✅ 显示 | ✅ 显示  | 两者同时展示，用户可选择 |
| 只有正价 | ✅ 显示 | ❌ 不显示 | 仅展示正价商品      |
| 只有临期 | ✅ 显示 | ✅ 显示  | 仅展示临期商品      |
| 都无库存 | ✅ 显示 | ❌ 不显示 | 仅展示正价商品（售罄）  |


**C端库存展示字段**:


| 字段   | 说明                |
| ---- | ----------------- |
| 可售库存 | 当前可购买的数量          |
| 库存状态 | 充足/库存紧张/已售罄（**保持现有阈值**[Q4]） |
| 效期信息 | 临期商品显示"效期至 XX-XX" |


**边界/异常**:

- 临期商品效期已过期 → 不展示
- 临期商品库存=0 → 展示"已售罄"状态

## 8. 非功能性要求


| 类别  | 要求                          | 原因       |
| --- | --------------------------- | -------- |
| 性能  | 库存同步延迟 ≤ 5秒（从WMS变化到C端展示）    | 用户体验     |

| 安全  | 库存数据不脱敏（内部系统间传输）            | -        |
| 审计  | 所有库存变动需记录操作日志（时间、类型、数量、订单号） | 追溯问题     |

## 9. 关键数据流

### 库存增加

```
┌─────────────────────────────────────────────────────────────┐
│  [正价商品]                                                  │
│  WMS入库 → 库存中心 ──────────→ 销售系统正价库存 → C端       │
│                                                             │
│  [临期商品]                                                  │
│  WMS临期标记 → WMS直连 ─────────→ 销售系统临期库存 → C端    │
│                          ↓                                  │
│                    扣减正价库存（库存中心通知）                              │
│                    （临期↑ = 正价↓）                         │
└─────────────────────────────────────────────────────────────┘
```

### 库存减少

```
┌─────────────────────────────────────────────────────────────┐
│  用户下单 → 销售系统预占（按FIFO）→ 预占指令（含批次）→ WMS  │
│                                                    ↓        │
│                              履约完成 → 扣减临期库存 ← WMS   │
│                                                    ↓        │
│                                            扣减临期库存      │
└─────────────────────────────────────────────────────────────┘
```

### C. 与现有系统差异说明（研发参考）

> 本章节列出PRD需求与现有代码实现的差异，供研发在做差异化开发或灰度控制时参考。

#### C1. 临期品同步方式差异

| 对比项 | 现有系统（SRM订单计划） | 本PRD需求 | 差异级别 |
|--------|------------------------|----------|---------|
| 同步触发 | 定时任务（10-30分钟间隔） | 实时推送 + 按需拉取 | **高** |
| 同步链路 | SRM → OMP → WMS | WMS直连 → 销售系统 | **高** |
| 相关Job | `NearExpirationDateSyncWmsJob` 等 | 新建实时链路 | 中 |

**研发建议**：
- 新旧链路并行运行，通过开关控制启用哪条
- 灰度方案：按SKU/仓库/时间段逐步切换

---

#### C2. 预占实现差异

| 对比项 | 现有系统（order-trade-service） | 本PRD需求 | 差异级别 |
|--------|-------------------------------|----------|---------|
| 预占Command | `ReserveStockCommandImpl` | 新建独立预占逻辑 | **高** |
| 预占时机 | 确认订单时 | 支付成功后 | 中 |
| 幂等保证 | `idempotentNo` | 需自行实现 | 中 |

**研发建议**：
- 评估是否复用 `ReserveStockCommandImpl`，降低重复开发
- 如复用，需扩展支持批次维度
- 如新建，需确保幂等性保证

---

#### C3. 临期品状态机差异

| 对比项 | 现有系统（SRM订单计划） | 本PRD需求 | 差异级别 |
|--------|------------------------|----------|---------|
| 状态定义 | PENDING → GENERATED → ABOARD/OFFBOARD → CANCELLED | 可售 → 已预占 → 待履约 → 已完成/已释放 | **高** |
| 状态驱动 | 人工/定时任务 | 系统自动（库存触发） | 高 |
| 状态存储 | `near_expiration_date_product_aboard` 表 | 新建 `expired_goods` 表 | 中 |

**研发建议**：
- 新旧状态机并行，通过商品类型区分
- 灰度方案：新品启用新状态机，老品逐步迁移

---

#### C4. 库存扣减时机差异

| 对比项 | 现有系统（loms-analysis Pipeline） | 本PRD需求 | 差异级别 |
|--------|----------------------------------|----------|---------|
| 扣减时机 | 解析第3阶段（分配成功）即扣减 | WMS履约完成通知后扣减 | **高** |
| Pipeline | 8阶段解析（预占→分配→出库） | 独立预占→WMS→履约完成 | 高 |
| 回滚机制 | finally块执行退预占 | 取消/失败时释放 | 中 |

**研发建议**：
- 新旧扣减时机不同，需明确以哪个为准
- 如并行，需在订单类型/商品类型上做路由

#### C5. 库存模型差异

| 对比项 | 现有系统（yh-oms-stock） | 本PRD需求 | 差异级别 |
